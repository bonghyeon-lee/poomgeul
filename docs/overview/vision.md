## 한 줄

**"공개된 텍스트를 누구나 번역하고, 여러 번역본이 동등하게 공존하는 오픈소스 허브."**

## 세 가지 축

1. **공개 텍스트 중심** — 소프트웨어 i18n이 아니라 논문·문학·기술문서 같은 산문.
2. **여러 번역본의 공존** — "정답 번역 하나"가 아니라 여러 fork가 동등한 자산으로 축적.
3. **AI 초벌 + 커뮤니티 교열** — LLM이 초안, 사람이 교열·분기·머지. GitHub 스타일 워크플로우.

## 원문은 위키, 번역본은 깃허브

poomgeul의 모든 UX 결정을 지배하는 비대칭.

- **원문(Source) = 위키 스타일** — 단일 버전으로 수렴. OCR 오류 수정·판본 정렬은 허용, force push 금지, 히스토리 투명.
- **번역본(Translation) = 깃허브 스타일** — 여러 번역본이 동등하게 공존. 큐레이터가 `featured`를 지정(M1+)하되, 모든 번역본은 자체 URL로 접근 가능.

### 왜 비대칭인가

- 원문을 fork하면 "이게 진짜 arXiv 논문이야?" 혼동이 생긴다. 신뢰의 근간이 흔들림.
- 번역본을 단일화하면 같은 문장의 미묘한 차이 — 바로 언어 학습의 본질 — 가 사라진다.

## Phase 1 타겟

| 순위 | 세그먼트 | 역할 | 핵심 니즈 |
|---|---|---|---|
| 1 | 영어 논문을 읽는 비영어권 대학원생·연구자 | 독자 | arXiv 논문을 모국어로 빠르게 이해 |
| 2 | 번역 포트폴리오를 쌓고 싶은 연구자·학생 | 기여자 | 기여 축적·피드백·가시성 |
| 3 | 언어 학습자 (학술 영어) | 독자 | 병렬 텍스트로 읽기 학습 |
| 4 | 본인 논문을 널리 알리고 싶은 저자 | 원문 제공자 | 번역 요청·기여 수집 |

- **언어 페어:** 영→한 1개 고정 (Phase 1).
- **문학·기타 언어:** Phase 2 이후.

> 페르소나 ② ③은 스케치 단계. Phase 0 인터뷰로 검증 예정. 결단 게이트는 기획서 §5.3.

## 핵심 가치 제안

- **기여자에게:** "번역이 커리어가 된다" — 프로필·크레딧·평판 축적.
- **독자에게:** "한 번역에 종속되지 않는다" (M1부터 완전 작동. M0는 병렬 뷰 + Attribution 투명성에 집중).
- **원저자에게:** "내 텍스트가 세계로 확장된다" — CC BY 합법 번역 + 자동 Attribution.

## Non-Goals

- 소프트웨어 i18n (Weblate·Tolgee 영역).
- Google Translate식 원클릭 번역.
- 상용 전문번역 서비스.
- 음성·영상 번역.
- 저작권 모호한 현대 문학·팬 번역.

## 포지셔닝 방어선 — 5가지 결합

단일 기능은 모두 대체 가능하지만, 아래 5가지의 결합은 어디에도 없습니다.

1. 공개 텍스트(산문) 대상
2. fork-and-PR 기반 협업
3. 여러 번역본의 공존 허용
4. CC/PD 라이선스 자동 검증
5. LLM 네이티브 초벌 엔진

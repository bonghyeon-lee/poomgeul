# 거버넌스 & 권한 모델

![phase](https://img.shields.io/badge/phase-M0%2B-green)

기획서 §11 발췌 + 구현 수준의 권한 매트릭스.

## 1. 3개 객체 × 각기 다른 권한

| 객체 | 수렴/분기 | 권한 모델 | 활성화 |
|---|---|---|---|
| 원문 | 수렴 (위키식) | 메인테이너가 편집, 이력 투명 | M0 |
| 번역본 | 분기 (깃허브식) | 번역본마다 리드 메인테이너 1명 + 제안 기반 기여 | M0 |
| 번역본 (다중 공존) | 분기 | 누구나 새 번역본 생성, fork 가능 | M1 |
| 메타데이터 | 공유 | 용어집·TM은 프로젝트 공동 자산 | M1+ |

> **활성화 원칙:** M0는 원문 1개 × 번역본 1개 구조. 번역본 내부 협업(제안/리뷰/머지)은 M0에서 작동하지만, "같은 원문에 여러 번역본이 공존"하는 차별점은 M1에서 열림. 스키마는 처음부터 다중 번역본 전제로 설계([data-model.md](../architecture/data-model.md)).

## 2. 원문 소유권 유형

**유형 A — 저자 생존·직접 등록** (CC BY 논문 저자 등)
- 저자 = 자동 메인테이너.
- 저자가 원문 수정 권한 보유.
- 번역본 관리는 위임하거나 자유 방임.

**유형 B — 퍼블릭 도메인 / 고아 저작물** (Phase 2)
- 플랫폼이 **큐레이터 역할**을 별도 설계.
- 예: "도스토옙스키 큐레이터", "1920년대 한국 단편 큐레이터".
- Wikisource 유사 모델.

**유형 C — 제3자 대리 등록**
- 등록자는 **기여자일 뿐, 자동 메인테이너가 아님**.
- 관리 권한은 평판으로 획득 (악용 방지).

## 3. 기여 워크플로우 (M0 기본 경로)

> **핵심 원칙:** "누구나 읽고, 누구나 제안하고, 메인테이너가 머지한다."

번역본마다 **리드 메인테이너 1명**을 두고, 제안/리뷰/머지 흐름으로 공동 작업을 받습니다. GitHub PR과 의도적으로 유사한 UX지만, 브랜치·리베이스·충돌 해결 같은 Git 개념 자체는 사용자에게 노출하지 않습니다.

상세 상태 머신은 [architecture/workflow-proposal.md](../architecture/workflow-proposal.md) 참조.

### M0에서 의도적으로 제외한 것

| 기능 | 상태 | 대체 |
|---|---|---|
| 다중 메인테이너 / 공동 소유 | M1 | 리드 메인테이너 1명 (TranslationCollaborator 테이블은 pre-design) |
| 투표·민주적 승인 | Phase 3+ | 리드 메인테이너 재량 |
| 4단계 티어 자동 승급 | M2 | 로그인 사용자 모두 제안 가능, 평판 계산 비활성 |
| 제안 간 머지/분기 | M2+ | 제안은 독립 단위, 동시 충돌은 optimistic locking으로만 |
| 라인 단위 inline 댓글 | M2 | 제안 전체에 대한 ProposalComment로만 |

## 4. 4단계 티어 시스템 (M2 활성화)

> **현재 (M0/M1):** 사용자 티어 = `로그인함 / 리드 메인테이너 / 어드민` 3단. 아래 4단계는 M2에서 활성화되는 장기 모델.

| 티어 | 권한 | 승급 기준 |
|---|---|---|
| **신규 (new)** | 문장 단위 번역 **제안**만 (승인 필요) | 가입 직후 — **M0 기본값** |
| **검증된 기여자 (verified)** | 제안 없이 직접 commit (해당 번역본 한정) | N개 이상 승인받은 기여 — **M2 활성화** |
| **메인테이너 (maintainer)** | 원문 구조 변경, 섹션 삭제, 번역본 승격 | 평판 임계값 또는 커뮤니티 선출 — **M1: 수동 지정, M2: 자동 승급** |
| **큐레이터·어드민 (curator/admin)** | 라이선스 분쟁 대응, 악용 대응, 원문 제거 | 코어팀 임명 → 점진적 커뮤니티 선출 — **M0부터 코어팀이 맡음** |

"누가 owner냐"를 정하지 않고, **권한을 행동 이력에 귀속**시키는 모델.

### 왜 M2까지 미루는가

자동 승급 로직은 평판 공식 튜닝(§11.6), 악용 탐지, 티어 간 알림/UX 차별화를 모두 필요로 합니다. 활성 사용자 수백 명 이하일 때는 메인테이너의 수동 판단이 빠르고 공정. §11.6 평판 공식은 M1부터 **데이터 수집 목적**으로 계산·표시만 합니다.

## 5. 미리 정해둘 설계 결정 (§11.5)

- **저자가 뒤늦게 나타날 때:** 원문 소유권 이관 프로세스를 미리 정의. "등록된 저자는 자신의 원문에 대한 메인테이너 권한을 요구할 수 있다" 조항.
- **원문 수정:** reversible-only. force push 금지. 모든 번역본이 영향을 받기 때문에 히스토리가 중요.
- **"fork"의 의미:** 원문 fork와 번역본 fork를 명확히 구분. 원문 fork는 정책적으로 제한, 번역본 fork는 자유(M1부터).
- **버려진(방치된) 번역:** 리드 메인테이너가 90일 이상 활동이 없으면 "계승 요청" 공개.
  - 옵션 1: 승계 (기존 번역본 소유권 이관)
  - 옵션 2: 독립 포크 (새 번역본으로 분기, M1부터)
  - 옵션 3: 공동 메인테이너 초대 (M1부터)
- **원문 삭제 요청:** CC BY는 법적으로 철회 불가하지만, 플랫폼 차원에서 "삭제 요청" 프로세스 설계 필요.
- **리드 메인테이너 사임·양도:** 본인이 언제든 다른 기여자에게 리드를 넘길 수 있음. 수신자 수락 시 즉시 이전. 사임만 하고 수신자 미지정이면 번역본이 "고아" 상태로 전환되어 계승 요청 대상이 됨.
- **제안 품질 악용 대응:** 대량 저품질 제안을 반복하는 사용자는 해당 번역본에서 제안 차단 + 전역 rate limit 대상. 코어팀이 수동 검토(M0~M1), 자동 탐지는 M2+.

## 6. 기여자 인센티브 (§11.6 요약)

기여자에게 "번역이 커리어가 된다"는 가치 제안을 실제 작동하게 만드는 요소. 단순 포인트/배지가 아니라 **외부 인용 가능성**과 **학술·커리어 신호**에 초점.

### 프로필 구성 요소
- 기여 통계: 번역 세그먼트 수, 승인률, 리뷰 세그먼트 수, 메인테이너 참여 번역본 수
- 번역 포트폴리오: featured 번역본 하이라이트
- 전문 분야 태그: arXiv 분류(cs.LG, cs.CL 등) 기반 자동 + 수동
- 공개 URL: `poomgeul.org/@username`

### 외부 연동
- **ORCID 연동** — 번역 기여를 ORCID 레코드에 `translation` 유형으로 푸시.
- **GitHub 연동** — 번역 활동을 GitHub 컨트리뷰션처럼 시각화.
- **DOI 발급** (Phase 2 검토) — featured 번역본에 Zenodo 연동.

### 공식 기여 증명
- 번역본 완료·featured 승격 시 **기여 증명서(PDF) 자동 발급** — 서명된 해시, 기여 범위, 승인자 명시.

### 평판 산정 공식 (제안)
```
score = Σ(approved_segments × segment_difficulty)
      + Σ(helpful_reviews × 0.5)
      + featured_bonus(translations × 20)
      - Σ(reverted_segments × 2)
```
- `segment_difficulty`는 세그먼트 길이·전문 용어 밀도로 자동 산정.
- 승급 임계값: verified ≥ 50, maintainer ≥ 500 + 커뮤니티/코어팀 승인.
- 구체 수치는 베타 6주차에 실제 데이터로 튜닝.

### 금전적 인센티브는 도입 안 함 (MVP)
- 번역 품질이 금전 유인에 의해 저품질 초벌 승인으로 왜곡될 위험.
- Wikipedia처럼 비금전 인정 시스템이 오래 가는 커뮤니티를 만듦.
- 스폰서 기반 "이 논문 번역에 $X bounty"는 Phase 3 이후 실험 여지.
